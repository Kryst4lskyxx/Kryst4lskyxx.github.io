## Why this is trickier than it looks

On Kubernetes the operator hides most of this: you shrink a `ClickHouseInstallation`, and the reconcile loop redraws the config and recreates pods. On **bare metal** there is no reconcile loop. "Move a shard out of the cluster" means *you* edit XML by hand, *you* move the data, and *you* sequence it so nothing is lost while a live Kafka stream keeps writing.

The two facts that trip people up:

1. ClickHouse **does not rebalance data.** Removing a shard from the config does not relocate a single row. The config controls *routing*, not *placement*.
2. The number you see as `shard_num` in `system.clusters` is **not** the `{shard}` macro that pins a node to its data. They are independent, and conflating them is how people accidentally orphan a replica's data.

This post is a source-grounded walk through both, plus the operational ordering for a Kafka-ingested cluster. All hostnames, cluster names, and shard numbers below are generic — the only thing taken verbatim is the open-source ClickHouse parsing code.

## The two numbers that are not the same number

### Number 1: the cluster shard number is *positional*

A cluster lives under `<remote_servers>` in a config file. Here is a two-shard, two-replica cluster:

```xml
<clickhouse>
  <remote_servers>
    <events_all>
      <shard>
        <internal_replication>true</internal_replication>
        <replica><host>ch-a01</host><port>9000</port></replica>
        <replica><host>ch-a02</host><port>9000</port></replica>
      </shard>
      <shard>
        <internal_replication>true</internal_replication>
        <replica><host>ch-b01</host><port>9000</port></replica>
        <replica><host>ch-b02</host><port>9000</port></replica>
      </shard>
    </events_all>
  </remote_servers>
</clickhouse>
```

When ClickHouse parses this, it numbers the shards by **counting them in document order**. From `src/Interpreters/Cluster.cpp`:

```cpp
// src/Interpreters/Cluster.cpp  (~line 447)
UInt32 current_shard_num = 1;

for (const auto & key : config_keys)
{
    bool shard_with_replicas = startsWith(key, "shard");
    bool shard_without_replicas = startsWith(key, "node");
    ...
    info.shard_num = current_shard_num;     // <-- the position, not a macro
    ...
    ++current_shard_num;                     // ~line 564
}
```

That is the whole story for `shard_num`: it is the *N*th `<shard>` element in the list. The parser never opens `macro.xml`. So the values in `system.clusters.shard_num` are *generated by counting* — they are always `1..N` and always contiguous, no matter what you write.

:::note
You *can* give shards explicit names with a `<name>` child, but it's all-or-nothing per cluster — `Cluster.cpp` throws `INVALID_SHARD_ID` if some shards have names and others don't (~line 440). Even then, the numeric `shard_num` is still positional.
:::

### Number 2: the `{shard}` macro pins a node to its *data*

Each host carries its own macros, usually in a per-host `config.d/macro.xml`:

```xml
<clickhouse>
  <macros>
    <shard>07</shard>
    <replica>ch-b01</replica>
  </macros>
</clickhouse>
```

These are plain string substitutions, handled in a completely different place (`src/Common/Macros.cpp`). They get expanded into **ReplicatedMergeTree ZooKeeper paths** and `ON CLUSTER` DDL — for example:

```sql
ENGINE = ReplicatedMergeTree('/clickhouse/tables/{shard}/events', '{replica}')
```

The ZK path is what makes two nodes *replicas of each other* (same `{shard}`, different `{replica}`) and what binds a node to the data it owns. The `<remote_servers>` block never reads it.

### So: do the macro and the shard definition need to be consecutive?

**No — and the macro value doesn't even need to match the position.** Because `shard_num` is generated by counting, and the macro is just a string in a ZK path, the two are free to diverge:

- The set of `{shard}` macro values does **not** have to be a contiguous `1..N`. You can skip `07` entirely; ClickHouse never validates the macro against anything.
- The macro value does **not** have to equal the node's position in the cluster. A node at position 7 can carry `{shard}` = `42`. Routing uses the position; data ownership uses the macro. Both work.

Keeping `position == macro` is a *convention* worth following for human sanity, but it is not a requirement the engine enforces.

## The membership multiplier (a bare-metal gotcha)

Before you delete anything, grep for how many times your nodes actually appear. Real clusters define **many** `<remote_servers>` entries over the same physical hosts — `events_all`, a `leader`/`replica` split, an `ssd`-only subset, plus per-tenant clusters. In one cluster I worked on, **every host appeared 48 times** across the config.

"Move the shard out" therefore means removing its `<shard>` block from *every* cluster you want it gone from — not just the obvious one. Find the blast radius first:

```bash
grep -c "ch-b01" cluster.xml        # how many clusters is this host in?
grep -nE "<[a-z_]+>|ch-b01" cluster.xml | grep -B1 "ch-b01"   # which ones
```

## Removing config never moves data

This is the load-bearing sentence of the whole operation:

:::warning
Deleting a `<shard>` block changes **routing only**. The data that lived on those nodes stays on those nodes' disks (and in their ZK paths). If that data is unique to the shard, dropping the shard out of the read path makes it *invisible*, and decommissioning the hardware *deletes it for good*. ClickHouse will not move it for you.
:::

Whether the data is unique depends on your write path. The common bare-metal pattern — and the one I'll use below — is **MV → local ReplicatedMergeTree**, where each node consumes a slice of the Kafka topic and writes to its *local* table. There, each shard holds rows nobody else has. You must either drain them or let them expire.

## Ingest first, or drain first?

If you're fed by the Kafka engine, the instinct "stop ingest, then drain, then remove" is correct — and the ordering is not cosmetic.

Draining means copying the shard's rows onto the remaining shards. If ingestion keeps writing to that shard *during or after* the copy, you've created a moving target: rows that land after your snapshot are stranded and lost when you drop the shard. **Freeze writes to the shard before the final drain.** Drain-then-stop loses the in-between window.

### How "stop ingest on one shard" actually behaves

The Kafka engine table on each node is part of a consumer group (`kafka_group_name`), and Kafka distributes the topic's partitions across *all* consumers in that group. The engine only consumes while a materialized view is attached and reading from it.

So to stop ingest on the target shard **only**:

1. `DETACH` the materialized view(s) on both of the shard's nodes — pauses the pull→push.
2. `DETACH TABLE` the Kafka engine table(s) — the consumers leave the group.

When those consumers leave, **Kafka rebalances their partitions onto the consumers on your remaining shards.** Cluster-wide ingestion continues with no topic-level loss — the stream that *would* have hit the dying shard now lands elsewhere. You never pause the whole cluster.

:::caution
After the rebalance, the other consumers resume from the **committed offsets**, so the most-recent messages the dying shard already consumed live *only* on that shard — they won't be re-read elsewhere. Keep the shard **queryable** until that tail is gone, or you'll lose a window of recent data.
:::

## The procedure (short-TTL realtime cluster)

Here's where the workload shape pays off. If your ingested tables have a **short TTL** (hours to days — common for realtime), you can let TTL do the draining and skip the manual copy entirely. The trick is to stop *writing* to the shard but keep *reading* from it until its data ages out.

**Phase 1 — Stop ingest on the target shard (both replicas)**

```sql
-- on each of the shard's two nodes
DETACH TABLE events_mv;        -- pause pull -> push
DETACH TABLE events_kafka;     -- leave the consumer group; Kafka rebalances
```

**Phase 2 — Let TTL drain it**

- Leave the shard **in `cluster.xml` and serving reads.** Its rows stay queryable through the Distributed table and expire naturally over `max(TTL)`.
- New data routes to the surviving shards via the Kafka rebalance — no forward gap.

**Phase 3 — Remove the shard**

- Once the shard's data has aged past TTL, delete its `<shard>` block from **every** relevant `<remote_servers>` cluster. `cluster.xml` is hot-reloaded — no restart.
- Drop the leftover tables and decommission the hardware.

If TTL is enforced only on merges, force the eviction before removal so you're not waiting on a background merge:

```sql
ALTER TABLE events MATERIALIZE TTL;
-- or, to be sure a shard is empty:
OPTIMIZE TABLE events FINAL;
```

**Verify at each step:**

```sql
-- consumers left here, picked up elsewhere
SELECT * FROM system.kafka_consumers;

-- on the dying shard, this should trend to zero
SELECT count(), max(event_time) FROM events;

-- run against the Distributed table; totals should stay smooth
SELECT count() FROM events_dist;
```

:::tip
If you can't wait out the TTL (you need the hardware back now) and can't tolerate that recent-data tail living only on the dying shard, fall back to an explicit drain after Phase 1: `INSERT INTO events_dist SELECT * FROM remote('ch-b01', currentDatabase(), events)` into the surviving shards before you remove it. With short TTL, waiting is simpler and lower-risk.
:::

## What positional renumbering does and doesn't break

Because `shard_num` is positional, deleting a shard from the *middle* of the list shifts every later shard's number down by one in `system.clusters`. What survives that:

- **ReplicatedMergeTree** — data binding is the `{shard}` macro / ZK path, untouched. Remaining nodes keep their macro and keep their data. *This is why you must never "close the gap" by renumbering the survivors' macros — that would point them at a different ZK path and orphan their data.*
- **Distributed reads** — routed by live position, work normally.

What to watch:

- **Distributed inserts with an explicit sharding key** — fewer shards changes the modulus, so the hash→shard mapping for *new* rows shifts. That redistribution is inherent to removing any shard, not a bug.
- **External tooling** that assumes `system.clusters.shard_num == {shard} macro` will now see them diverge for every shard after the hole. Cosmetic to ClickHouse, but it can confuse dashboards and scripts.

## Takeaways

- The cluster `shard_num` is **positional and auto-counted** (`Cluster.cpp`); the `{shard}` **macro** is a string in a ZK path. They are independent — macros need not be consecutive, nor match position.
- **Config controls routing, not placement.** Removing a `<shard>` block moves zero rows. Drain (or expire) the data yourself first.
- For Kafka ingestion, **stop consumers on the target shard only** — Kafka rebalances its partitions to the rest, and the cluster keeps ingesting.
- **Order is load-bearing:** stop ingest → let data settle/expire (or copy it) → remove from config → decommission.
- With **short TTL**, keep the shard readable until its data ages out and skip the manual copy — but never yank it from the read path before the recent tail is gone.
- Find the **full membership** of your nodes before editing; one host is usually in many clusters.

The bare-metal version of this work is mostly discipline: the engine is happy to let you orphan data, so the safety lives entirely in the sequencing.
